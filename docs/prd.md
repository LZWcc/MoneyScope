# MoneyScope MVP PRD

## Problem Statement

MoneyScope 当前已经完成课程项目的仓库骨架、需求文档、数据库设计和小组分工说明，但应用代码仍处于占位状态。作为成员 A，核心问题是需要把项目从“文档初始化”推进到“可运行、可演示、可解释”的 MVP，同时保持代码结构清晰，方便其他组员后续体验、截图、整理报告和录制演示。

课程项目还要求体现 Python 技术应用、SQLite 数据持久化、面向对象设计、异常处理、AI 使用记录、Git 协作记录和完整提交材料。因此 MVP 不能只做一个简单页面，而要形成从数据录入、持久化、筛选、统计、图表到导入导出的完整闭环。

## Solution

本阶段交付一个基于 Python、Streamlit、SQLite、pandas 和 plotly 的个人消费记录与数据分析 MVP。用户可以在中文界面中添加收入和支出记录，查看并筛选交易明细，查看月度收入、支出和结余，分析分类支出占比和月度趋势，并通过 CSV 导入导出准备演示数据。

系统采用分层结构：数据库层负责 SQLite 建表和连接，模型层定义交易与预算对象，服务层封装业务逻辑和统计计算，图表层生成可视化对象，Streamlit 入口只负责页面展示与交互。这样既符合课程对面向对象和清晰结构的要求，也方便成员 A 解释关键代码。

## User Stories

1. As a 普通用户, I want 添加收入记录, so that 我可以保存生活费、兼职收入或其他收入来源。
2. As a 普通用户, I want 添加支出记录, so that 我可以记录餐饮、交通、学习、娱乐等日常消费。
3. As a 普通用户, I want 输入交易日期, so that 我的记录可以按月份统计和筛选。
4. As a 普通用户, I want 选择收入或支出类型, so that 系统可以正确计算收入、支出和结余。
5. As a 普通用户, I want 输入交易分类, so that 我可以分析不同类别的消费情况。
6. As a 普通用户, I want 输入金额并得到校验提示, so that 我不会保存小于等于 0 或格式错误的数据。
7. As a 普通用户, I want 输入备注, so that 我可以回忆这笔交易的具体原因。
8. As a 普通用户, I want 查看全部交易记录, so that 我可以确认数据已经被保存。
9. As a 普通用户, I want 按月份筛选记录, so that 我可以只查看某个月的收支。
10. As a 普通用户, I want 按收入或支出筛选记录, so that 我可以分别查看不同类型的数据。
11. As a 普通用户, I want 按分类筛选记录, so that 我可以快速定位某类消费。
12. As a 普通用户, I want 删除错误记录, so that 我可以修正误录入的数据。
13. As a 普通用户, I want 查看月度总收入, so that 我了解当月资金来源。
14. As a 普通用户, I want 查看月度总支出, so that 我了解当月消费规模。
15. As a 普通用户, I want 查看月度结余, so that 我判断自己是否超支。
16. As a 普通用户, I want 查看分类支出占比图, so that 我知道钱主要花在哪些方面。
17. As a 普通用户, I want 查看月度趋势图, so that 我可以观察收入和支出的变化。
18. As a 普通用户, I want 设置总预算或分类预算, so that 我可以提前规划消费上限。
19. As a 普通用户, I want 在接近或超过预算时看到提示, so that 我可以及时调整消费。
20. As a 普通用户, I want 导入 CSV 示例数据, so that 我可以快速准备课堂演示数据。
21. As a 普通用户, I want 导入错误 CSV 时看到中文错误提示, so that 我知道应该修改哪些字段。
22. As a 普通用户, I want 导出交易记录 CSV, so that 我可以备份数据或用于报告材料。
23. As a 成员 A, I want 业务逻辑集中在服务层, so that Streamlit 页面不会混入 SQL 和复杂统计逻辑。
24. As a 成员 A, I want 数据库 SQL 集中在数据库层, so that 表结构和连接错误容易维护。
25. As a 成员 A, I want 至少两个有意义的类, so that 项目满足面向对象设计要求。
26. As a 成员 A, I want 关键函数有 docstring, so that 答辩时可以解释函数职责。
27. As a 成员 A, I want 基础测试覆盖服务层, so that 后续修改时能快速发现回归。
28. As a 成员 B, I want 运行后能直接体验核心功能, so that 我可以反馈页面和功能问题。
29. As a 成员 C, I want 有稳定的演示数据和页面, so that 我可以整理截图。
30. As a 成员 D, I want README 和测试记录与实际功能一致, so that 课程提交材料可信。
31. As a 成员 E, I want 功能路径简单清楚, so that 演示视频可以顺利录制。

## Implementation Decisions

- 使用 Python 3.10+、Streamlit、SQLite、pandas 和 plotly，不引入 Flask、FastAPI、Django、独立前端框架或云数据库。
- MVP 保持本地单用户应用，不实现登录、多用户、云同步或复杂推荐。
- 数据库使用 SQLite，本地数据库文件位于 `data/demo_full.db`，仓库只提交示例 CSV 和建表代码，不提交真实消费数据库。
- 核心表包括交易记录、分类和预算。MVP 中交易分类先保存为文本，降低实现和答辩复杂度。
- `Transaction` 表示一条收入或支出记录，`Budget` 表示月度总预算或分类预算。
- Streamlit 页面使用中文文案，负责表单、筛选控件、摘要卡片、图表和错误提示展示。
- 服务层暴露稳定接口用于新增、删除、查询、统计、预算检查、CSV 导入和 CSV 导出。
- 图表层只接收统计后的数据并返回 plotly 图表对象，不直接访问数据库。
- CSV 导入字段固定为 `date,type,category,amount,description`，缺字段、日期错误、类型错误和金额错误都要返回清晰中文错误。
- 预算提醒采用简单规则：支出达到预算 80% 时提醒接近预算，超过预算时提醒已超预算。
- AI 使用记录在开发过程中补充到 `AI_LOG.md`，不在最后集中虚构。

## Testing Decisions

- 测试优先覆盖服务层，因为服务层是数据持久化、校验、统计和 CSV 行为的核心边界。
- 好测试应该验证外部行为，例如新增记录后能查到、无效金额会报错、统计结果正确、错误 CSV 会被拒绝，而不是依赖函数内部实现细节。
- 数据库测试使用临时 SQLite 文件，避免污染真实 `data/demo_full.db`。
- 统计测试使用少量固定交易数据，验证收入、支出、结余、分类汇总和趋势数据。
- CSV 测试覆盖有效导入、缺少必填字段、非法金额和导出字段顺序。
- Streamlit 页面以手动测试为主，在 `docs/test-report.md` 中记录添加、筛选、图表、预算、导入导出等演示路径。

## Out of Scope

- 用户登录、权限管理和多账户体系。
- 云端同步、远程数据库和多人实时协作。
- 机器学习推荐、复杂消费预测和自动记账。
- 微信、支付宝或银行账单自动解析。
- 复杂仪表盘、大规模 UI 重构或独立前端项目。
- 保存真实个人隐私消费数据到仓库。
